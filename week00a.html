The Map of the Territory

This journal follows six connected parts of the LLM lifecycle: data, pretraining, post-training, evaluation, inference, and operations. Today I start with the smallest working version of the core generation loop.

  1. What does a GPT-style model produce before the decoder chooses the next token?
  2. What are the two broad training phases between raw text and a useful assistant?
  3. Why is prompt prefill usually more compute-intensive, while autoregressive decode is often memory-bandwidth-bound even though it emits only one token per step?

I have used language models in production, but I want a more exact picture of what happens between raw text and a served system. For the next six months I am building small, inspectable versions of each part and publishing the code, measurements, failures, and corrections.

This is the map I am starting with. I expect parts of it to change as I work through the residency.

The core generation loop

The GPT-style autoregressive models this journal focuses on take the tokens seen so far and produce one score, or logit, for every token in their vocabulary. The decoding procedure turns those scores into next-token probabilities, chooses a token, appends it, and repeats.

The toy model below skips the neural network and stores probabilities directly. That makes the loop visible without pretending the model is a miniature ChatGPT.

That is the core generation loop, not the whole product. Tokenization, data, training, post-training, evaluation, serving infrastructure, retrieval, tools, and product policy all shape what users experience.

Build one right now

The source dataset contains 32,033 name records and 29,494 unique spellings. The lab deduplicates the records, reserves validation and test data before fitting the model, and learns which character tends to follow which. It runs on a laptop CPU.

Open the lab in Colab → The notebook includes the held-out evaluation and self-checks. The compact version below trains and samples the same model.

Python 
from pathlib import Path
import hashlib, random, urllib.request
import torch

# Pin the source so the data cannot change underneath this example.
url = ("https://raw.githubusercontent.com/karpathy/makemore/"
       "988aa59e4d8fefa526d06f3b453ad116258398d4/names.txt")
path = Path("names-988aa59.txt")
if not path.exists():
    urllib.request.urlretrieve(url, path)

data = path.read_bytes()
expected = "0a30b5557f192f32ab962680889aac5f6fda0f4cecf40a6d0b5694f58ea8cc4d"
assert hashlib.sha256(data).hexdigest() == expected
records = data.decode("utf-8").splitlines() # 32,033 records
words = sorted(set(records))                  # 29,494 unique spellings
random.Random(42).shuffle(words)
n_train = int(0.8 * len(words))
n_val = int(0.9 * len(words))
train_words = words[:n_train]                 # held-out splits stay untouched
val_words = words[n_train:n_val]
test_words = words[n_val:]

chars = sorted(set("".join(train_words)))
stoi = {s: i + 1 for i, s in enumerate(chars)}
stoi["."] = 0
itos = {i: s for s, i in stoi.items()}

# Fit P(next character | current character) on training names only.
N = torch.zeros((27, 27), dtype=torch.int32)
for w in train_words:
    cs = ["."] + list(w) + ["."]
    for c1, c2 in zip(cs, cs[1:]):
        N[stoi[c1], stoi[c2]] += 1

P = (N + 1).float()                          # add-one smoothing
P /= P.sum(1, keepdim=True)

g = torch.Generator().manual_seed(42)
for _ in range(5):
    out, ix = [], 0
    while True:
        ix = torch.multinomial(P[ix], 1, generator=g).item()
        if ix == 0:
            break
        out.append(itos[ix])
    print("".join(out))

With PyTorch 2.13 on CPU, seed 42 begins with a, feeenvi, s, mabian, and dan. The exact samples are recorded with the environment because PyTorch does not guarantee identical seeded results across every release and platform.

  • It learned signal. Uniform guessing scores 3.2958 nats per transition. This model scores 2.4537 on training data, 2.4567 on validation data, and 2.4606 on test data. The small gap suggests its one-character transition statistics transfer similarly to held-out spellings; it does not, by itself, rule out memorization.
  • Its representation is weak. One character of context is too little, so every transition can be locally likely while the complete name is implausible. The model can generate novel strings, but it can also reproduce training names—dan is one example here. Sampling is not a novelty guarantee.

This toy exercise touches data preparation, model fitting, held-out evaluation, and inference. It omits post-training and operations, and reduces the model itself to a table. Those omissions are the rest of the route.

The six parts of the lifecycle

This is the order I use to organize the journal, not a one-way production pipeline. Evaluation runs throughout, and problems found in operations feed back into data, training, and deployment.

01Data

“The internet” is not a dataset. Web crawls contain boilerplate, duplicates, spam, low-quality text, and material that may need exclusion for privacy, licensing, safety, or policy reasons. Filtering, deduplication, and dataset mixture can materially change quality and memorization. In Week 7 I will run a small pipeline and measure those effects instead of treating “more data” as the answer.

02Pretraining

For a GPT-style model, pretraining repeatedly asks a transformer to predict the next token and adjusts its parameters when it is wrong. Large runs can involve trillions of tokens and billions of parameters; some frontier runs are estimated to cost tens to hundreds of millions of dollars. In Weeks 1–8 I will do this on one GPU—enough to encounter core optimization failures such as unstable loss and bad learning-rate schedules, but not distributed-training failures.

03Post-training

A pretrained model is a completion engine, not an assistant. Ask it “What’s the capital of France?” and it may reply “What’s the capital of Germany? What’s the capital of Italy?”, because quiz lists look like that on the internet. One common post-training route uses supervised fine-tuning on demonstrations, followed by preference optimization. In Weeks 9–12 I will use those methods to turn the base model into something conversational and measure what regresses.

04Evals

How do I know any of it worked? Benchmark items can appear in training data, scores can depend on prompt format, and stochastic generation or model-based judges introduce variance. A score without its dataset, protocol, and uncertainty can be misleading. In Week 8, then again in Week 20, I will build an evaluation harness with held-out data, repeat stochastic measurements, and report uncertainty rather than trusting one number.

05Inference

Training is episodic; inference runs on every request. Prompt prefill is usually more compute-intensive, while autoregressive decode at low or moderate batch sizes is often constrained by memory bandwidth. The KV cache is a major consumer of serving memory and a central scheduling constraint. Weeks 13–17 cover prefill, decode, KV-cache sizing, batching, quantization, speculative decoding, and GPU cost accounting.

06Operations

Models are software in production: dashboards, latency targets, load spikes, incidents at 2 a.m. A serving fleet can be up and still be failing: KV-cache pressure can trigger preemption, recomputation, and latency spikes; quantization can regress the task-specific evaluation that matters. Running models is a reliability discipline. In Week 18 I will deliberately break the serving stack and drill the recovery.

Why I spend six weeks on serving

My background is systems and reliability, and inference is the part I most want to understand deeply. Weeks 13–18 therefore form a six-week inference-and-operations block: decoding, KV-cache pressure, batching, quantization, latency, cost, and operating the resulting service. The earlier foundations stay necessary because serving behavior only makes sense when I understand the model being served.

Sources & further reading

  • Bengio et al., A Neural Probabilistic Language Model (2003): neural next-token modeling and longer context.
  • Karpathy, makemore: the code and pinned dataset behind the Week 0 lab.
  • Shannon, Prediction and Entropy of Printed English (1951): human next-letter prediction as a way to estimate the entropy of English.
  • Lee et al., Deduplicating Training Data Makes Language Models Better (2021): data duplication, memorization, and evaluation overlap.
  • Ouyang et al., Training language models to follow instructions with human feedback (2022): supervised fine-tuning followed by preference-based post-training.
  • Zhong et al., DistServe (OSDI 2024): the different latency and resource behavior of prefill and decode.
  • Kwon et al., Efficient Memory Management for Large Language Model Serving with PagedAttention (2023): KV-cache memory and serving throughput.
  • Cottier et al., The rising costs of training frontier AI models (2024): estimated training and development costs.
